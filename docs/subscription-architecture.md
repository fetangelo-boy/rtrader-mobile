# RTrader — Архитектура системы заявок на подписку

## 1. Summary

Предлагается **единая модель заявок** (`subscription_requests`), в которую попадают все обращения на подписку — независимо от канала (Telegram или Email). Бизнес-логика одна: заявка создаётся, администратор проверяет чек, вручную задаёт дату окончания подписки, после чего система автоматически создаёт или продлевает доступ и уведомляет пользователя через тот же канал, откуда пришла заявка.

Ключевые принципы:

- **Канал — это только вход.** Telegram и Email — два адаптера к одному и тому же subscription flow.
- **Администратор решает всё.** Никакой автоматики после получения чека. Дата окончания задаётся вручную.
- **Доступ — ровно до указанной даты.** Не "+30 дней", а конкретная дата, выбранная администратором.
- **Один пользователь — один аккаунт.** Система различает новых и существующих по `telegram_id` или `email`.

## 2. Сущности и таблицы

### 2.1 Текущие таблицы (без изменений)

| Таблица | Назначение |
|---------|-----------|
| `auth.users` (Supabase Auth) | Аккаунты пользователей, `user_metadata` хранит `telegram_id` |
| `profiles` | Профиль: `id`, `username`, `avatar_url`, `created_at` |
| `subscriptions` | Подписки: `id`, `user_id`, `plan`, `status`, `started_at`, `expires_at` |
| `chats`, `messages`, `chat_participants` | Чаты и сообщения |

### 2.2 Новая таблица: `subscription_requests`

Центральная сущность — **заявка на подписку**. Одна таблица для обоих каналов.

```
subscription_requests
├── id              UUID (PK, auto-generated)
├── channel         TEXT ('telegram' | 'email')
├── status          TEXT ('pending_review' | 'approved' | 'rejected' | 'executed' | 'failed')
│
├── telegram_id     TEXT (nullable, заполняется для Telegram-заявок)
├── telegram_name   TEXT (nullable, имя из Telegram)
├── email           TEXT (nullable, заполняется для Email-заявок)
├── contact_name    TEXT (nullable, имя из email-формы)
│
├── receipt_url     TEXT (URL скриншота чека в S3)
├── receipt_text    TEXT (nullable, комментарий от пользователя)
│
├── admin_note      TEXT (nullable, комментарий администратора)
├── approved_until  TIMESTAMPTZ (nullable, дата окончания подписки — задаётся администратором)
├── approved_plan   TEXT (nullable, тарифный план — задаётся администратором)
├── approved_by     TEXT (nullable, ID администратора)
│
├── user_id         UUID (nullable, FK → auth.users.id, заполняется после execute)
├── is_new_user     BOOLEAN (nullable, true если создан новый аккаунт)
│
├── executed_at     TIMESTAMPTZ (nullable, когда доступ был выдан)
├── error_message   TEXT (nullable, если execute не удался)
│
├── created_at      TIMESTAMPTZ (default now())
├── updated_at      TIMESTAMPTZ (default now())
```

### 2.3 Жизненный цикл заявки

```
                    ┌─── Telegram-бот ───┐
                    │                    │
Пользователь ──────►  subscription_requests  ◄────── Email-форма
                    │   status: pending_review      │
                    └────────────┬───────────────────┘
                                 │
                    ┌────────────▼───────────────────┐
                    │   Администратор проверяет чек   │
                    │   Задаёт approved_until          │
                    │   Задаёт approved_plan            │
                    └────────────┬───────────────────┘
                                 │
                         ┌───────┴───────┐
                         │               │
                    approved         rejected
                         │               │
                         ▼               ▼
                    execute()        Уведомление
                    ┌────────┐      "Заявка отклонена"
                    │ Новый? │
                    ├───yes──┤
                    │ create │
                    │ user + │
                    │ sub    │
                    ├───no───┤
                    │ renew  │
                    │ sub    │
                    └────┬───┘
                         │
                    executed
                         │
                    Уведомление
                    через тот же канал
```

### 2.4 Статусы заявки

| Статус | Описание | Кто устанавливает |
|--------|----------|-------------------|
| `pending_review` | Заявка создана, ожидает проверки администратором | Система (автоматически при создании) |
| `approved` | Администратор одобрил, задал дату окончания | Администратор |
| `rejected` | Администратор отклонил заявку | Администратор |
| `executed` | Доступ выдан/продлён, пользователь уведомлён | Система (после успешного execute) |
| `failed` | Ошибка при выдаче доступа (например, Supabase API упал) | Система (при ошибке execute) |

## 3. API-эндпоинты

### 3.1 Создание заявки (публичные, без admin key)

Эти эндпоинты вызываются Telegram-ботом или email-формой. Они **не требуют** `X-Admin-Key`, потому что не выдают доступ — только создают заявку.

**`POST /api/requests/create`**

```json
{
  "channel": "telegram",
  "telegram_id": "123456789",
  "telegram_name": "Иван Петров",
  "receipt_url": "https://s3.../receipt.jpg",
  "receipt_text": "Оплата за месяц"
}
```

или для email:

```json
{
  "channel": "email",
  "email": "ivan@example.com",
  "contact_name": "Иван Петров",
  "receipt_url": "https://s3.../receipt.jpg",
  "receipt_text": "Оплата за месяц"
}
```

Ответ:

```json
{
  "success": true,
  "request_id": "uuid",
  "status": "pending_review",
  "message": "Ваш чек принят и находится на рассмотрении администратором, пожалуйста, ожидайте."
}
```

### 3.2 Загрузка чека (публичный)

**`POST /api/requests/upload-receipt`** (multipart/form-data)

Загружает скриншот чека в S3 и возвращает URL. Используется перед `create`.

### 3.3 Админ-эндпоинты (требуют X-Admin-Key)

**`GET /api/admin/requests`** — Список заявок

```
GET /api/admin/requests?status=pending_review&limit=20&offset=0
```

Ответ: массив заявок с полной информацией.

**`GET /api/admin/requests/:id`** — Детали одной заявки

**`POST /api/admin/requests/:id/approve`** — Одобрить заявку

```json
{
  "approved_until": "2026-06-01T00:00:00Z",
  "approved_plan": "premium",
  "admin_note": "Чек проверен, оплата подтверждена"
}
```

Этот эндпоинт:
1. Устанавливает `status = 'approved'`, `approved_until`, `approved_plan`
2. Автоматически вызывает `execute()`:
   - Ищет пользователя по `telegram_id` или `email`
   - Если не найден → создаёт аккаунт (Supabase Auth + profile + subscription)
   - Если найден → обновляет `subscriptions.expires_at` до `approved_until`
3. Устанавливает `status = 'executed'` или `status = 'failed'`
4. Возвращает результат (логин/пароль для нового, подтверждение для существующего)

```json
{
  "success": true,
  "request_id": "uuid",
  "status": "executed",
  "is_new_user": true,
  "user_id": "uuid",
  "email": "tg123456789@rtrader.app",
  "password": "Xk9mP2qR4v",
  "subscription": {
    "plan": "premium",
    "expires_at": "2026-06-01T00:00:00Z"
  }
}
```

Бот или email-система получает этот ответ и отправляет уведомление пользователю.

**`POST /api/admin/requests/:id/reject`** — Отклонить заявку

```json
{
  "admin_note": "Чек не соответствует сумме"
}
```

### 3.4 Существующие admin-эндпоинты (остаются)

Текущие эндпоинты (`create-subscriber`, `renew-subscription`, `reset-password`, `block-subscriber`, `subscriber-status`) остаются как **прямые команды** для экстренных случаев, когда администратору нужно действовать без заявки.

## 5. UX: Администратор

### 5.1 Вариант A — Через Telegram-бота (MVP)

Администратор работает прямо в Telegram:

1. Бот пересылает заявку в админ-чат (фото чека + данные пользователя)
2. Администратор отвечает командой:

```
/approve <request_id> 2026-06-01
```

или

```
/reject <request_id> Чек не соответствует
```

3. Бот вызывает Admin API и пересылает результат пользователю.

**Преимущества:** Не нужен отдельный интерфейс. Администратор работает в том же Telegram.

### 5.2 Вариант B — Админ-панель в приложении (после MVP)

Отдельный экран в приложении (доступен только для `role: admin`):

- Список заявок с фильтрами (pending / approved / rejected)
- Карточка заявки: фото чека, данные пользователя, история
- Кнопки "Одобрить" (с выбором даты) и "Отклонить" (с комментарием)
- Статистика: сколько заявок в очереди, средний срок рассмотрения

### 5.3 Рекомендация

**MVP: Вариант A** (Telegram-бот). Администратор уже в Telegram, заявки приходят туда же. Минимум разработки. Админ-панель можно добавить позже.

## 6. Как система различает новых и существующих

При выполнении `execute()` после одобрения заявки:

```
1. Извлечь telegram_id или email из заявки
2. Поиск в auth.users:
   - По telegram_id: user_metadata.telegram_id === telegram_id
   - По email: user.email === email
3. Если найден:
   → is_new_user = false
   → UPDATE subscriptions SET expires_at = approved_until
   → Уведомление: "Подписка продлена до {дата}"
4. Если не найден:
   → is_new_user = true
   → CREATE user (Supabase Auth)
   → CREATE profile
   → CREATE subscription (expires_at = approved_until)
   → Сгенерировать пароль
   → Уведомление: "Логин: {email}, Пароль: {password}, До: {дата}"
```

| Задача | Оценка |

## 9. Альтернативное предложение: Invite-код с ручным одобрением

Если хочется ещё проще и элегантнее, можно совместить заявку и invite-код:

1. Пользователь отправляет чек (Telegram или Email)
2. Администратор проверяет, одобряет, задаёт дату
3. Система генерирует **одноразовый invite-код** (например, `RTR-A7K9-M2PX`)
4. Код отправляется пользователю
5. Пользователь вводит код в приложении на экране логина
6. Приложение:
   - Если код валиден и пользователь новый → показывает форму регистрации (задать пароль)
   - Если код валиден и пользователь существующий → автоматически продлевает подписку

**Преимущества:**
- Пользователь сам выбирает пароль (не нужно запоминать сгенерированный)
- Код можно передать любым способом (Telegram, email, SMS, голосом)
- Не нужно отправлять пароли в открытом виде
- Один механизм для всех каналов

**Недостатки:**
- Лишний шаг для пользователя (ввести код)
- Нужен экран ввода кода в приложении

Этот вариант можно реализовать **параллельно** с основным (Вариант A), как дополнительный способ выдачи доступа.
